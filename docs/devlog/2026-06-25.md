## Duplicate Trip: Copy Reservations and Documents

### Problem
The Duplicate trip feature was missing reservations and attachments/documents. User was relying on Duplicate as a backup mechanism and discovered the copies were incomplete.

### What Was Done

1. **`src/lib/server/storage.ts`** — Added `copyObject()` using S3 `CopyObjectCommand` to server-side copy Spaces objects without re-downloading them.

2. **`src/lib/server/clone.ts`** — Extended `duplicateTrip` with two new sections:
   - **Reservations**: copies all reservation rows with id remapping for attachment FK references.
   - **Attachments**: copies all attachment rows (both `file` and `text` kinds). File attachments get a new S3 object via `copyObject` with a fresh random key under the new trip's prefix. Text documents are pure row copies. All FKs (reservation_id, itinerary_item_id) are remapped through the existing maps.

## Day Plans Enhancement: Adversarial Review Fixes

Reviewed the uncommitted day-plan enhancement diff after Claude Code's phase 6 pass and fixed several edge cases.

### Fixes

1. **Driving leg integrity**
   - `computeLegDistances()` now rejects routes unless every visible stop has coordinates, avoiding misleading "from previous" labels when an unlocated stop sits between two located stops.
   - Saved-plan "Calculate distances" now only enables when every stop has coordinates.
   - Persisted driving summaries now require every stop after the first to have `drive_km`/`drive_min`; partial stored legs no longer show as complete totals.
   - `bulkUpdateDriving()` now requires submitted leg destination IDs to exactly match the current saved stop order after the first stop.
   - `parseDrivingLegs()` now bounds submitted km/min values to reject absurd or overflow-prone data.

2. **Suggestion action side effects**
   - `dayplan-add-suggestion` now validates the day plan before creating any external itinerary item, preventing orphan places on bad `plan_id`.
   - External suggestions now require valid latitude/longitude before creating an itinerary item.
   - `dayplan-suggest` now requires at least two coordinate-bearing stops on the server, matching the UI route-context requirement.

3. **Weather date matching**
   - Cached NWS weather periods now retain `startTime`/`endTime` and keep up to 14 periods.
   - Day-plan weather display now prefers periods matching the plan's optional date, falling back to near-term daytime periods when the plan date is outside the forecast window.

4. **Suggestion context**
   - Internal and external stop suggestions now include and display approximate distance from the route centroid.
   - The "Suggest stops" button now disables unless the saved plan has at least two coordinate-bearing stops.
