# 2026-06-27

## Day Plan Safari save fix + Google place IDs

Fixed the Day Plan builder Safari regression and restored the saved-stops collapse toggle. The builder submit path now writes `title`, `optional_date`, `notes`, and `stops` into `FormData` from Svelte state and waits for `update({ reset: false })` before clearing local builder state. Verified with macOS Safari via `safaridriver`: a three-stop plan saved in one click with correct title/date/notes/stops, the `Stops` disclosure toggled, and Optimize Order completed.

Fixed Google Maps place links at the data layer instead of falling back to unsafe name-only searches. New location saves now carry a Google `place_id` through `/api/geocode`, `MapPicker`, manual location save, itinerary import, Google Maps URL extraction, and Day Plan external suggestions. Manual location saves now reject coordinate-only submissions rather than silently saving a pin that cannot open the real Google place detail panel. Day Plan stops continue to snapshot `itinerary_items.place_id`.

Added `scripts/backfill-place-ids.mjs`, a dry-run-first production backfill/audit tool. Ran it against production trip `Maine 2026` only:
- scanned 49 located places missing `place_id`
- updated 37 confident Google place ID matches
- skipped 12 low-confidence or too-far candidates
- left Day Plan snapshots untouched per request
- copied the apply audit CSV to `data/place-id-backfill/2026-06-27T19-57-29-418Z-apply-place-id-backfill.csv` locally
