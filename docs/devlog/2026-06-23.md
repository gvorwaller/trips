# 2026-06-23 Devlog

## Freeform Itinerary Import With Preview + Geocoding

### Summary
Added an LLM-assisted itinerary import flow that accepts rough pasted travel text, extracts structured itinerary candidates, lets the owner review/edit/select them, and imports the selected items under a chosen itinerary parent.

### What Was Done

1. **Freeform itinerary extraction**
   - Added `src/lib/server/itinerary-extract.ts`.
   - Uses the Anthropic Messages API with a forced `submit_itinerary_candidates` tool call.
   - Accepts rough prose, bullets, headings, and model output instead of requiring user-supplied JSON.
   - Returns candidate `place`, `day`, `section`, and `note` items with notes, URLs, address/location hints, optional source coordinates, and children.
   - Normalizes output defensively: max 200 items, max depth 3, valid item types, valid dates, valid coordinate ranges, trimmed field lengths.

2. **Review-before-import UI**
   - Added an "Import itinerary from text" details panel in the Places section.
   - Extract button is disabled until text is present and shows an extracting state during the LLM call.
   - Preview rows are editable before save: checkbox, item type, title, date, notes, address, location query, and reference URL.
   - Added Select all / Select none controls, import count, parent dropdown, and Geocode places checkbox.
   - Duplicate-looking candidates are flagged server-side and default to unselected when returned by the extractor.

3. **Transactional import path**
   - Added `src/lib/server/itinerary-import.ts`.
   - Re-validates all selected candidates server-side.
   - Verifies the optional parent belongs to the same trip.
   - Inserts selected candidates transactionally while preserving hierarchy and sibling order.
   - Geocodes places when enabled and stores coordinates directly on imported itinerary items.
   - If geocoding is disabled or fails, address/location-query text is preserved in notes so location hints are not lost.

4. **Trip page actions**
   - Added `itin-extract` action: extracts candidates only, with no database writes.
   - Added `itin-import-candidates` action: imports selected reviewed candidates.
   - Existing quick "Paste many" flow remains unchanged.

5. **Help and planning docs**
   - Updated Help with the rough-text import workflow and review-before-import behavior.
   - Added `docs/2026-06-23-json-itinerary-import-plan.md` documenting the design shift from user-supplied JSON to freeform text extraction.

## Itinerary Import Double-Submit Guard + Named Map Links

### Summary
Follow-up from production testing of the freeform itinerary import. The user saw duplicate-looking imported sections in the Maine 2026 trip and also noticed that place links opened Google/Apple Maps as raw coordinate pins instead of named places.

### Evidence Gathered

1. **Duplicate import root cause**
   - Production nginx access logs showed two successful import writes from Safari:
     - `2026-06-23 18:31:53 +0000` — `POST /trips/1?/itin-import-candidates` → `200`
     - `2026-06-23 18:31:59 +0000` — `POST /trips/1?/itin-import-candidates` → `200`
   - Production DB timestamps matched exactly:
     - First batch at `2026-06-23 18:31:53.456895+00`
     - Second partial duplicate batch at `2026-06-23 18:31:59.800696+00`
   - This points to two import submissions, not one extractor result duplicating itself.
   - The second duplicate batch appeared to be itinerary item IDs `24-33`.

2. **Weaknesses identified**
   - The client cleared import candidates after a successful import, but the Import button did not have an immediate in-flight guard.
   - The server flagged possible duplicates during extraction preview, but the import action did not re-check duplicates at write time.
   - Concurrent or repeated import requests could therefore insert the same candidate payload twice.

### Code Changes

1. **Client double-submit protection**
   - Updated `src/routes/trips/[id]/+page.svelte`.
   - Added `itinImporting` state.
   - `importSelectedItinerary()` now returns immediately if an import is already running.
   - The Import button disables while the request is in flight and changes label to `Importing...`.
   - Candidates and pasted text still clear after a successful import.

2. **Server-side duplicate guard**
   - Updated `src/lib/server/itinerary-import.ts`.
   - Import now takes a per-trip PostgreSQL advisory transaction lock before writing.
   - Inside the transaction, it refreshes current itinerary titles for the trip.
   - It skips duplicate-looking candidate titles using the same normalized-title style as preview duplicate detection.
   - The guard also catches duplicates within the same submitted candidate payload.

3. **Named place map links**
   - Updated `src/lib/maplinks.ts`.
   - `Google` and `Apple` place browsing links now prefer a name query when a title exists, even if coordinates are saved.
   - Coordinate-only links remain the fallback when no name exists.
   - The dedicated `Directions` link still uses coordinates when available so navigation remains precise.

4. **Tests**
   - Updated `src/lib/maplinks.test.ts`.
   - Added/adjusted assertions that:
     - Google place browsing uses `query=Arborvine` even when coordinates exist.
     - Google falls back to coordinates only when no name exists.
     - Apple Maps uses `q=...` for named places and `ll=...` only for coordinate-only entries.
     - Google directions still targets coordinates when available.
